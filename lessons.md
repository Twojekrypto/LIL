# LIL Project – Lessons Learned

## Workflow
- Daily update workflow: `/daily-update` — fetch on-chain data, update calculator, record history
- `fetch_onchain.js` scans ~21k blockchain chunks (EB Protocol + EB Staking), takes ~30 min
- Always verify 100% coverage (discovered vs on-chain totals) before committing

## Data Architecture
- `data/embed_data.js` — auto-generated by `fetch_onchain.js`, contains `LOCK_DATA`, `STAKE_DATA`, `PORTFOLIO_DATA`
- `index.html` embeds LOCK_DATA and STAKE_DATA directly (lines ~2269-2271), PORTFOLIO_DATA is computed in JS
- `data/history.json` — daily entries with staked/locked/burned/stakers/lockers counts
- Calculator reads totals from `history.json` via fetch, with hardcoded fallback

## Update Checklist (index.html)
1. Replace `LOCK_DATA` and `STAKE_DATA` arrays from fresh `embed_data.js`
2. Update `AVG_LOCK_MULTIPLIER` (weighted avg from LOCK_DATA lock durations using LOCK_POINTS interpolation)
3. Update circulation stats: Locked/Staked total + percentage
4. Update disclaimer date in calculator info-bar
5. Update `ON-CHAIN DATA (fetched ...)` comment
6. Append entry to `data/history.json`

## Design Patterns
- Summary cards use `circ-stats` grid with `circ-stat` items (colored borders, progress bars)
- Tables use `holders-tbl` with `renderGenericTable` for pagination
- Badge colors: blue=locked, amber/green=staked, purple=team, cyan=total
- `EFFECTIVE_SUPPLY = 986,812,159` (post-burn)

## Common Pitfalls
- `totalBurned` is hardcoded (363,187,841) — update only if new burn events happen
- AVG_LOCK_MULTIPLIER uses interpolated multiplier from LOCK_POINTS, NOT raw `days/365`
- The `embed_data.js` file is NOT loaded by index.html — data is copy-pasted into HTML
